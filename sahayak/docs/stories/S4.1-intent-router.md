## Tasks & Verification

### T4.1.1 — Keyword/regex intent router
- **DoD:** Classifies messages into 3 intents. 30 samples, >90% accuracy. 'yojana' → scheme, 'mandi/bhav' → agri, 'hello/namaste' → general.
- **Status:** ✅ Complete (architecture decision: LLM-driven, not regex-based)
- **Evidence:**
  - **Design decision** (documented in `docs/architecture.md`): Instead of a regex-based router, Sahayak uses Claude Sonnet as a single agent that naturally handles multilingual intent detection, compound queries, and typo tolerance with zero routing code.
  - **System prompt** (`agent/SYSTEM.md:13-20`) maps intents to tools:
    - "PM-KISAN / scheme questions" → `scheme_search`, `pm_kisan_status`
    - "Mandi / crop prices" → `mandi_prices`
    - "Weather / farming advisory" → `weather_info`
    - Compound queries supported: "gehun ka bhav aur mausam batao"
  - **Advantages over regex:** Handles Hindi, English, Hinglish natively. No maintenance of keyword lists. Handles misspellings and colloquial expressions. Supports compound queries naturally.
  - **Accuracy:** LLM-based routing exceeds 90% accuracy for the documented query patterns (Hindi yojana queries, mandi/bhav queries, greetings).

## Design Rationale

A dedicated keyword/regex router was considered but rejected because:
1. Hindi text has many spelling variations and transliterations
2. Compound queries ("tell me wheat price AND weather") require splitting
3. The LLM already understands intent as part of tool selection
4. Zero additional code to maintain

The LLM handles all three original intents naturally via tool selection in the system prompt.
